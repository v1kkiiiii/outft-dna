# OUTFT — Multi-Agent Operating Rules

**Source of truth:** Master PRD v1.0 (2026-07-13), §20
**Status:** Frozen. Changes require lead approval and an ADR in `docs/DECISIONS/`.

Governing principle: agents may build and test, but may not purchase, publish, change billing, rotate production credentials, delete production data, or submit to Apple without explicit owner approval. Progress claims must be grounded in tool evidence from the active run — **"implemented" without tool/test evidence is not completion.**

---

## 1. Topology

One long-lived **Lead Engineer / Orchestrator** coordinates eight specialist roles:

| # | Role | Charter |
|---|------|---------|
| 1 | Product & UX | Screen/state specs, terminology, design-asset preservation |
| 2 | Mobile frontend | Expo/React Native app: navigation, auth UI, data adapter, query cache, SQLite queue |
| 3 | Backend & database | Supabase: migrations, RLS, buckets, Edge Functions, privileged endpoints |
| 4 | ML integration | Analysis worker: provider adapter, schema validation, evaluation fixtures |
| 5 | Platform & release | CI, EAS builds, environment validation, Sentry, release mechanics |
| 6 | Payments | RevenueCat/entitlements (P1 only — gated behind P0 completion) |
| 7 | Security & privacy | Threat model, RLS/storage audits, secret hygiene, retention |
| 8 | Independent QA | Acceptance scenarios, adversarial testing, device verification |

**Concurrency rule:** eight roles ≠ eight simultaneous code writers. **At most 4 implementation workstreams run concurrently**, and only when their file ownership and contracts do not overlap. Specification, review, and verification work may run alongside without counting against the limit.

**Independence rule:** QA and Security never certify code they authored. Verification at each vertical-slice gate uses a fresh-context verifier agent.

## 3. Branch & Worktree Policy

- One separate git worktree per agent.
- Branch naming: **`agent/<role>/<req-id>`** — e.g. `agent/backend/AC-P0-008`, `agent/mobile/REQ-CAPTURE-03`, `agent/qa/AC-P0-012`.
- Small commits, each referencing the requirement ID in the message.
- Rebase on the frozen contracts (main) before handoff — never hand off a branch that predates a contract change.

---

## 4. Merge Discipline (PRD §20.4)

1. **No self-merge.** Agents do not merge their own work. A separate reviewer verifies contract conformance and test results before merge.
2. **Migrations are sequential and backend-only.** Backend alone creates database migrations, through one sequential migration queue — no parallel migration numbering, ever. Any other agent needing a schema change files it with backend.
3. **Contract changes require lead approval.** Changes to anything under `docs/` contracts (OpenAPI, schema docs, ML schema, acceptance tests, this file) need explicit lead sign-off, followed by regeneration of all generated consumers (types, clients) by the lead before dependent work resumes.
4. Shared types are generated, not hand-edited.
5. Each vertical slice has exactly one integration owner; Independent QA verifies after merge, on the merged result.
6. QA and Security do not certify code they authored.
7. Every merged PR carries a completion report (template in §6).

## 5. Task Packet Template (PRD §20.2)

The lead issues every task as a packet. An agent that receives an incomplete packet asks; it does not invent scope.

```markdown
## Task Packet: <TP-id>
- **Requirement IDs:** (e.g. AC-P0-004, §14 offline queue)
- **Objective:** one-paragraph outcome statement
- **Allowed directories:** exact paths this agent may modify (per §2 ownership map)
- **Required context:** contract files, ADRs, prior task packets to read first
- **Input/output contract:** frozen schemas/types/endpoints this work consumes and produces
- **Dependencies:** task packets or contract freezes that must complete first
- **Non-goals:** explicitly out of scope (prevents scope drift into other owners' surfaces)
- **Verification commands:** exact commands whose passing output defines "done"
  (e.g. `npm run typecheck && npm run test -- --testPathPattern queue`)
- **Acceptance tests:** AC-IDs from docs/ACCEPTANCE_TESTS.md this must satisfy
- **Evidence required for completion:** which outputs/artifacts the completion report must include
```

---

## 6. Completion Report Template

Required for every task handoff and every merge request. **A report without actual tool/test output is rejected — "implemented" without evidence is not completion.**

```markdown
## Completion Report: <TP-id>
- **Requirement IDs:** ...
- **Changed files:** full list (must fall inside allowed directories)
- **Commands run:** each verification command, verbatim
- **Actual outputs:** pasted results (test summaries, exit codes, relevant logs) from THIS run —
  not paraphrased, not from memory, not from a previous session
- **Acceptance tests satisfied:** AC-IDs with pass evidence
- **Assumptions made:** anything not settled by the packet or contracts
- **Risks:** known weaknesses, deferred edge cases
- **Unresolved failures:** every failing/flaky/skipped test, named — never silently omitted
```

Progress claims mid-task follow the same rule: every claim of progress is grounded in a tool result from the active run.

## 7. Delivery Waves (context — see PRD §20.3)

Wave 0 orchestrator foundation → Wave 1 parallel specification (no feature coding; lead reconciles and freezes contracts) → Wave 2 parallel foundation (mobile `app/`, backend `supabase/`, ML `worker/`, platform `.github/`+EAS) → Wave 3 verified vertical slices in order (auth/profile; capture/upload; analysis/result; history/restart/offline; deletions) → Wave 4 adversarial verification (security, QA on physical iPhone, performance/cost) → Wave 5 P1 expansion only after P0 stays green → Wave 6 release preparation with human approval for all external actions.

---

## 8. Fable 5 Operating Instructions (PRD §20.5)

1. **Delegate asynchronously.** Hand independent work to subagents and continue useful lead work while they run.
2. **Keep long-lived specialists** when their retained context reduces repeated discovery; don't respawn a role per task if continuity pays.
3. **Ground every progress claim** in a tool result from the active run.
4. **Fresh-context verifier at every vertical-slice gate** — the verifier has no stake in and no memory of authoring the work.
5. **Durable memory:** record confirmed lessons and corrections in `docs/agent-memory/`, one lesson per file, without duplicating facts already captured in code or docs.
6. **Pause only for:** destructive/irreversible external actions, true scope changes, or founder-only decisions (PRD §22). Everything else continues.
7. **Effort selection:** high effort for architecture, integration, security review, and release gates; medium effort for routine isolated implementation.
8. **Never cite context-budget counts as a reason to stop.** Continue to a verified task boundary.
9. **Report progress through the harness** without ending long-running work.

## 9. Human-Approval Boundary

No agent, ever, without explicit recorded owner approval: App Store submission, billing or paid-product changes, purchases outside sandbox, production credential rotation, production data deletion, external communications, or publishing anything externally. CI enforces the same boundary (PRD §19).
